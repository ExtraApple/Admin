# 登录安全策略实施文档

> **文档定位**: 已实现模块的实现说明。当前行为以 `openspec/specs/auth/spec.md` 为准；本文保留实现细节、数据流和测试步骤。

## 当前状态

登录安全策略已完整实现，包括 **密码复杂度校验、验证码保护、渐进式登录锁定** 三大机制。

## 模块目标

登录安全策略负责在登录环节提供多层安全防护，防止暴力破解和撞库攻击。

最终效果：

```text
用户登录请求
  → 验证码校验（防机器人）
  → 锁定检查（Redis lock:<username>）
  → 查找用户 + 状态检查
  → 密码比对
    ├─ 失败 → 失败计数 +1 → 达到阈值则锁定
    └─ 成功 → 清除所有失败记录 → 签发 JWT
```

## 功能清单

| 功能 | 状态 | 说明 |
|---|---|---|
| 密码复杂度校验 | ✅ | 最少6位，3种字符类型 |
| 验证码保护 | ✅ | 6位数字图片验证码，Redis 5分钟过期，一次性消费 |
| 密码重试限制 | ✅ | 5次错误后开始锁定 |
| 渐进式锁定 | ✅ | 失败次数越多，锁定时间越长 |
| 失败记录自动过期 | ✅ | fail 计数 24 小时过期，lock 标记按梯度过期 |
| 登录成功清除记录 | ✅ | 成功后立即清除 fail + lock key |
| bcrypt 加密 | ✅ | 密码永不明文存储 |

## Redis 存储设计

| Key | 类型 | 用途 | 过期策略 |
|---|---|---|---|
| `captcha:<id>` | string | 验证码答案 | 5 分钟自动过期 |
| `fail:<username>` | int | 登录失败次数计数器 | 24 小时自动过期 |
| `lock:<username>` | string | 账号锁定标记 | 按失败次数梯度设置 |

```go

    store := &redisStore{}
    return store.Verify(id, answer, true) // 无论成功失败都删除
```

说明：

- 每个验证码只能使用一次
- 即使验证失败也会删除，防止暴力穷举验证码
- 每次登录/注册都需要重新获取验证码

## 密码规则

### 规则定义

- 最少 6 位
- 必须包含大写字母、小写字母、数字、特殊符号中至少 3 种

### 实现

```go
// service/user_helper.go

func validatePassword(pw string) error {
    if len(pw) < 6 {
        return errors.New("密码长度不能少于 6 位")
    }

    var hasUpper, hasLower, hasDigit, hasSpecial bool
    for _, r := range pw {
        switch {
        case unicode.IsUpper(r):
            hasUpper = true
        case unicode.IsLower(r):
            hasLower = true
        case unicode.IsDigit(r):
            hasDigit = true
        case unicode.IsPunct(r) || unicode.IsSymbol(r):
            hasSpecial = true
        }
    }

    count := 0
    for _, ok := range []bool{hasUpper, hasLower, hasDigit, hasSpecial} {
        if ok { count++ }
    }
    if count < 3 {
        return errors.New("密码必须包含大写字母、小写字母、数字、特殊符号中至少 3 种")
    }
    return nil
}
```

应用场景：

- 注册（Register）
- 修改密码（ChangePassword）

## 渐进式锁定规则

### 锁定梯度

| 失败次数 | 锁定时间 | 说明 |
|---|---|---|
| 1–4 | 不锁定 | 提示剩余尝试次数 |
| 5–9 | 1 分钟 | 第 5 次开始锁定 |
| 10–14 | 5 分钟 | 锁定后清除失败计数，重新累计 |
| 15–19 | 15 分钟 | — |
| 20+ | 60 分钟（1 小时） | 最高级别锁定 |

### 实现

```go
// service/user_helper.go

const maxFailures = 5

func lockDuration(failures int) int {
    switch {
    case failures < maxFailures:  // 0-4
        return 0
    case failures < 10:           // 5-9
        return 1
    case failures < 15:           // 10-14
        return 5
    case failures < 20:           // 15-19
        return 15
    default:                       // 20+
        return 60
    }
}
```

### 锁定流程

```go
func lockAfterFail(username string, failures int) {
    dur := lockDuration(failures)
    if dur == 0 {
        return // 未达阈值，不锁定
    }

    // 设置锁定标记，过期时间 = 锁定分钟数
    global.Redis.Set(ctx, "lock:"+username, "1", time.Duration(dur)*time.Minute)

    // 达到 10 次时清除 fail 计数（防止计数器无限增长）
    if failures >= 10 {
        global.Redis.Del(ctx, "fail:"+username)
    }
}
```

## 登录完整流程

```text
POST /api/login
  │
  ├─ 1. 验证码校验
  │   └─ 失败 → 400 "验证码错误或已过期"（验证码已删除，需重新获取）
  │
  ├─ 2. 锁定检查
  │   └─ Redis.exists("lock:<username>") → 返回 TTL 告知剩余锁定时间
  │
  ├─ 3. 查找用户
  │   └─ 不存在 → 400 "用户名或密码错误"（模糊提示）
  │
  ├─ 4. 状态检查
  │   └─ status != 1 → 400 "账号已被禁用"
  │
  ├─ 5. 密码比对
  │   ├─ 失败 → incr fail:<username>
  │   │         → lockAfterFail(username, count)
  │   │         → 400 "用户名或密码错误（剩余尝试: N 次）"
  │   │
  │   └─ 成功 → clearFailed(username)  // 删除 fail + lock key
  │            → 查询 roles + permissions
  │            → 签发 JWT
  │            → 200 "登录成功"
```

## 数据流

```text
┌─────────────┐
│ GET /captcha │
└──────┬──────┘
       │ 生成 6 位数字
       │ Redis SET captcha:<uuid> = "482917" EX 300
       │ 返回 base64 PNG
       ▼

┌──────────────┐
│ POST /login   │
└──────┬───────┘
       │
       ├── Redis GET captcha:<id> → 比对 → DEL captcha:<id>
       │   └─ 失败 → 400（验证码已删除）
       │
       ├── Redis EXISTS lock:<username>
       │   └─ 存在 → 400 "账号已被锁定，请 N 分钟后重试"
       │
       ├── MySQL SELECT user WHERE username = ?
       │   └─ 不存在 → 400（模糊提示）
       │
       ├── status != 1 → 400 "账号已被禁用"
       │
       └── bcrypt.CompareHashAndPassword
           ├─ 失败 → Redis INCR fail:<username>
           │         → lockDuration(count) > 0 ?
           │           YES → Redis SET lock:<username> = "1" EX dur
           │            NO → 400 "剩余尝试: N 次"
           │
           └─ 成功 → Redis DEL fail:<username> lock:<username>
                    → SQL JOIN 查询 roles + permissions
                    → 签发 JWT → 200
```

## 安全约束

| 机制 | 实现方式 |
|---|---|
| 密码不明文存储 | bcrypt 加密，salt 内嵌 |
| 验证码一次性 | 校验后立即删除（无论成功失败） |
| 验证码过期 | Redis TTL 5 分钟 |
| 暴力破解防护 | 渐进式锁定，5 次起锁 |
| 失败计数过期 | Redis TTL 24 小时，避免永久锁定 |
| 登录成功清理 | 立即清除所有失败记录和锁定标记 |
| 模糊错误提示 | 不区分"用户不存在"和"密码错误" |

## Apifox 测试顺序

1. 获取验证码（GET /api/captcha）
2. 用正确验证码 + 错误密码登录 → 提示"剩余尝试: 4 次"
3. 连续错误 5 次 → 提示"账号已被锁定，请 1 分钟后重试"
4. 等待 1 分钟后重新获取验证码，尝试登录
5. 继续错误 → 触发 5 分钟锁定
6. 锁定期间用正确密码也无法登录 → 验证锁生效
7. 锁定过期后，用正确密码登录 → 成功，验证记录被清除
8. 再次登录成功 → 验证无残留锁定

## 注意事项

- `fail:<username>` 和 `lock:<username>` key 基于用户名而非 IP，防止攻击者切换 IP 绕过
- 验证码校验失败时也会删除验证码（true），防止一个验证码被反复尝试
- 锁定达到 10 次失败时清除 fail 计数器，重新累计，实现梯度升级
- 验证码图片参数（倾斜系数 0.7、噪点 80）在平衡 OCR 抗性和人眼可读性之间做了折中
- 所有 Redis key 均设置 TTL，不会无限膨胀
- 密码校验规则同时应用于注册和修改密码两个场景

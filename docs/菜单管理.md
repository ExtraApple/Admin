# 菜单管理实施文档

## 当前建议

下一步建议实现 **菜单管理**。

原因很简单：用户、角色、权限已经基本成型，后台系统接下来需要让前端根据用户身份动态渲染侧边栏菜单。菜单管理完成后，`个人中心-初始上下文` 接口里的 `menus` 就可以从空数组升级为真实菜单树。

## 模块目标

菜单管理负责维护后台左侧导航、页面路由和角色可见菜单之间的关系。

最终效果：

```text
用户登录
  -> 查询用户角色
  -> 查询角色绑定的菜单
  -> 返回菜单树
  -> 前端渲染左侧导航
```

## 数据模型设计

### model/menu.go

```go
type Menu struct {
    gorm.Model
    ParentID  uint   `gorm:"index;comment:父级菜单ID"`
    Name      string `gorm:"type:varchar(100);not null;comment:菜单名称"`
    Path      string `gorm:"type:varchar(255);comment:前端路由路径"`
    Component string `gorm:"type:varchar(255);comment:前端组件路径"`
    Icon      string `gorm:"type:varchar(100);comment:菜单图标"`
    Sort      int    `gorm:"type:int;default:0;comment:排序"`
    Type      int    `gorm:"type:tinyint;default:1;comment:类型 1目录 2菜单 3按钮"`
    Status    int    `gorm:"type:tinyint;default:1;comment:状态 1启用 0禁用"`
}
```

```go
type RoleMenu struct {
```

## DTO 设计

### request/menu.go

```go
type CreateMenuReq struct {
    ParentID  uint   `json:"parent_id"`
    Name      string `json:"name" binding:"required,min=1,max=100"`
    Path      string `json:"path" binding:"max=255"`
    Component string `json:"component" binding:"max=255"`
    Icon      string `json:"icon" binding:"max=100"`
    Sort      int    `json:"sort"`
    Type      int    `json:"type"`
    Status    int    `json:"status"`
}

type UpdateMenuReq struct {
    ParentID  *uint  `json:"parent_id"`
    Name      string `json:"name" binding:"max=100"`
    Path      string `json:"path" binding:"max=255"`
    Component string `json:"component" binding:"max=255"`
    Icon      string `json:"icon" binding:"max=100"`
    Sort      *int   `json:"sort"`
    Type      *int   `json:"type"`
    Status    *int   `json:"status"`
}

type MenuInfo struct {
    ID        uint       `json:"id"`
    ParentID  uint       `json:"parent_id"`
    Name      string     `json:"name"`
    Path      string     `json:"path"`
    Component string     `json:"component"`
    Icon      string     `json:"icon"`
    Sort      int        `json:"sort"`
    Type      int        `json:"type"`
    Status    int        `json:"status"`
    Children  []MenuInfo `json:"children,omitempty"`
}

type AssignMenusToRoleReq struct {
    MenuIDs []uint `json:"menu_ids" binding:"required"`
}
```

## 接口设计

所有接口都在管理员路由组下：

```text
/api/admin
```

需要：

```text
Authorization: Bearer <admin_token>
```

### 1. 获取菜单树

```http
GET /api/admin/menus
Authorization: Bearer <管理员token>
```

返回：

```json
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "parent_id": 0,
      "name": "系统管理",
      "path": "/system",
      "component": "Layout",
      "icon": "settings",
      "sort": 1,
      "type": 1,
      "status": 1,
      "children": [
        {
          "id": 2,
          "parent_id": 1,
          "name": "用户管理",
          "path": "/system/users",
          "component": "system/users/index",
          "icon": "users",
          "sort": 1,
          "type": 2,
          "status": 1
        }
      ]
    }
  ]
}
```

### 2. 创建菜单

```http
POST /api/admin/menus
Content-Type: application/json
Authorization: Bearer <管理员token>
```

Body：

```json
{
  "parent_id": 0,
  "name": "系统管理",
  "path": "/system",
  "component": "Layout",
  "icon": "settings",
  "sort": 1,
  "type": 1,
  "status": 1
}
```

### 3. 修改菜单

```http
PUT /api/admin/menus/:id
Content-Type: application/json
Authorization: Bearer <管理员token>
```

Body：

```json
{
  "name": "系统设置",
  "icon": "settings",
  "sort": 2,
  "status": 1
}
```

### 4. 删除菜单

```http
DELETE /api/admin/menus/:id
Authorization: Bearer <管理员token>
```

约束：

- 如果菜单有子菜单，不允许直接删除
- 删除菜单时同步删除 `role_menus` 中的关联记录

### 5. 角色绑定菜单

```http
POST /api/admin/roles/:id/menus
Content-Type: application/json
Authorization: Bearer <管理员token>
```

Body：

```json
{
  "menu_ids": [1, 2, 3]
}
```

说明：

- `:id` 是角色 ID
- `menu_ids` 是菜单 ID 数组
- 采用全量替换：先清空该角色已有菜单，再写入新菜单

### 6. 获取角色菜单

```http
GET /api/admin/roles/:id/menus
Authorization: Bearer <管理员token>
```

返回该角色绑定的菜单树。

### 7. 同步前端路由到菜单表

```http
POST /api/admin/menus/sync
Content-Type: application/json
Authorization: Bearer <管理员token>
```

Body：

```json
{
  "routes": [
    {
      "name": "用户管理",
      "path": "/system/users",
      "component": "system/users/index",
      "icon": "users",
      "parent_path": "/system",
      "sort": 1
    }
  ]
}
```

说明：

- 前端把路由表提交给后端
- 后端根据 `path` 判断是否已存在
- 不存在则创建，存在则跳过或更新基础信息


建议方法：

```go

CreateMenu
UpdateMenu

```

### 5. 更新路由

在 `router/router.go` 的管理员路由组中加入：

```go
menuHandler := &handler.MenuHandler{}

admin.GET("/menus", menuHandler.ListMenus)
admin.POST("/menus", menuHandler.CreateMenu)
admin.PUT("/menus/:id", menuHandler.UpdateMenu)
admin.DELETE("/menus/:id", menuHandler.DeleteMenu)
admin.POST("/menus/sync", menuHandler.SyncMenus)
admin.POST("/roles/:id/menus", menuHandler.AssignRoleMenus)
admin.GET("/roles/:id/menus", menuHandler.GetRoleMenus)
```

### 6. 更新初始上下文接口

当前：

```json
"menus": []
```

```go
menus, _ := service.GetUserMenus(userID)
```

最终 `/api/user/context` 返回：

```json
{
  "user": {},
  "roles": [],
  "permissions": [],
  "menus": []
}
```

## 数据流

```text
用户登录
  -> JWT 注入 userID
  -> /api/user/context
  -> 查询 user_roles
  -> 查询 role_menus
  -> 查询 menus
  -> 组装菜单树
  -> 返回给前端
```

## 权限关系

菜单管理不直接替代权限管理。

推荐关系：

```text
菜单控制“看不看得到”
权限控制“能不能执行”
```

例子：

```text
editor 角色：
  菜单：文章管理
  权限：article:list, article:create, article:update
```

前端显示文章管理菜单，但删除按钮仍然需要 `article:delete` 权限才显示。

## Apifox 测试顺序

1. 创建一级菜单：系统管理
2. 创建二级菜单：用户管理
3. 获取菜单树
4. 创建角色或使用已有角色
5. 给角色绑定菜单
6. 获取角色菜单
7. 使用该角色用户登录
8. 调 `/api/user/context` 查看菜单是否返回

## 注意事项

- 菜单删除前必须检查是否存在子菜单
- 菜单树必须按 `sort asc, id asc` 排序
- `path` 建议唯一，方便同步前端路由
- `component` 只给前端使用，后端不解析
- 菜单和权限是两套系统，但应配合使用

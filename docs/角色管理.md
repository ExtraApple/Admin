# 角色管理实施文档

> **文档定位**: 已实现模块的实现说明。当前角色、用户角色、角色权限相关行为以 `openspec/specs/rbac/spec.md` 为准；本文保留模型、接口和实现步骤说明。

## 当前状态

角色管理已完成全部 CRUD 及关联功能，是 RBAC 权限体系的核心纽带——上连用户，下接权限和菜单。

## 模块目标

角色管理负责维护系统中的角色定义、角色与用户的关联、角色与权限的关联、角色与菜单的关联。

最终效果：

```text
用户登录
  → 查询 user_roles（用户绑定了哪些角色）
  → 查询 role_permissions（角色绑定了哪些权限码）
  → 查询 role_menus（角色绑定了哪些菜单）
  → 注入 JWT（roles + permissions）
  → 前端根据 roles 判断路由权限，根据 permissions 控制按钮显隐
```

## 数据模型设计

### model/role.go

```go
type Role struct {
    gorm.Model
    Name        string `gorm:"type:varchar(50);not null;uniqueIndex;comment:角色名称"`
    Code        string `gorm:"type:varchar(50);not null;uniqueIndex;comment:角色编码"`
    Description string `gorm:"type:varchar(255);comment:角色描述"`
    Sort        int    `gorm:"type:int;default:0;comment:排序"`
    Status      int    `gorm:"type:tinyint;default:1;comment:状态 1启用 0禁用"`
}

// UserRole 用户-角色关联表
type UserRole struct {
    UserID uint `gorm:"primaryKey;comment:用户ID"`
    RoleID uint `gorm:"primaryKey;comment:角色ID"`
}
```

### model/role_menu.go (在 model/menu.go)

```go
type RoleMenu struct {
    RoleID uint `gorm:"primaryKey;comment:角色ID"`
    MenuID uint `gorm:"primaryKey;comment:菜单ID"`
}
```

### model/role_permission.go (在 model/permission.go)

```go
type RolePermission struct {
    RoleID       uint `gorm:"primaryKey;comment:角色ID"`
```

## DTO 设计

### request/role.go

```go
// ========== 角色请求参数 ==========

type CreateRoleReq struct {
    Name        string `json:"name"        binding:"required,min=1,max=50"`
    Code        string `json:"code"        binding:"required,min=1,max=50"`
    Description string `json:"description" binding:"max=255"`
    Sort        int    `json:"sort"`
    Status      int    `json:"status"`
}

type UpdateRoleReq struct {
    Name        string `json:"name"        binding:"max=50"`
    Code        string `json:"code"        binding:"max=50"`
    Description string `json:"description" binding:"max=255"`
    Sort        *int   `json:"sort"`
    Status      *int   `json:"status"`
}

type AssignUsersToRoleReq struct {
    UserIDs []uint `json:"user_ids" binding:"required"`
}

// ========== 角色响应 ==========

type RoleInfo struct {
    ID          uint   `json:"id"`
    Name        string `json:"name"`
    Code        string `json:"code"`
    Description string `json:"description"`
    Sort        int    `json:"sort"`
    Status      int    `json:"status"`
}

type RoleListResp struct {
    List  []RoleInfo `json:"list"`
    Total int64      `json:"total"`
    Page  int        `json:"page"`
    Size  int        `json:"size"`
}
```

## 接口设计

所有接口在管理员路由组下：

```text
/api/admin
```

需要：

```text
Authorization: Bearer <admin_token>
```

### 1. 获取角色列表

```http
GET /api/admin/roles?page=1&size=10
Authorization: Bearer <admin_token>
```

返回：

```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": 1,
        "name": "超级管理员",
        "code": "admin",
        "description": "系统超级管理员",
        "sort": 1,
        "status": 1
      },
      {
        "id": 2,
        "name": "编辑者",
        "code": "editor",
        "description": "内容编辑角色",
        "sort": 2,
        "status": 1
      }
    ],
    "total": 2,
    "page": 1,
    "size": 10
  }
}
```

### 2. 创建角色

```http
POST /api/admin/roles
Content-Type: application/json
Authorization: Bearer <admin_token>
```

Body：

```json
{
  "name": "编辑者",
  "code": "editor",
  "description": "内容编辑角色",
  "sort": 2,
  "status": 1
}
```

约束：

- `name` 和 `code` 必须唯一
- status 默认 1（启用）

返回：

```json
{
  "code": 200,
  "msg": "创建成功",
  "data": {
    "id": 2,
    "name": "编辑者",
    "code": "editor",
    "description": "内容编辑角色",
    "sort": 2,
    "status": 1
  }
}
```

### 3. 修改角色

```http
PUT /api/admin/roles/2
Content-Type: application/json
Authorization: Bearer <admin_token>
```

Body：

```json
{
  "name": "高级编辑者",
  "description": "高级内容编辑角色",
  "sort": 3
}
```

约束：

- `code="admin"` 的角色不可修改
- 修改 name/code 时进行唯一性校验（排除自己）

### 4. 删除角色

```http
DELETE /api/admin/roles/2
Authorization: Bearer <admin_token>
```

约束：

- `code="admin"` 的角色不可删除
- 硬删除角色本身
- 自动清除 `user_roles` 中的关联记录

### 5. 分配用户到角色

```http
POST /api/admin/roles/2/users
Content-Type: application/json
Authorization: Bearer <admin_token>
```

Body：

```json
{
  "user_ids": [3, 5, 7]
}
```

说明：

- `:id` 是角色 ID
- 采用全量替换：先清空该角色已有用户，再写入新用户列表
- 传空数组 `[]` = 清空该角色下所有用户

### 6. 查看角色下的用户

```http
GET /api/admin/roles/2/users
Authorization: Bearer <admin_token>
```

返回该角色绑定的所有用户列表（脱敏）。

### 7. 分配权限到角色

```http
POST /api/admin/roles/2/permissions
Content-Type: application/json
Authorization: Bearer <admin_token>
```

Body：

```json
{
  "permission_ids": [1, 2, 3, 5]
}
```

说明：

- 全量替换 `role_permissions` 表
- 传空数组 = 清空该角色所有权限

### 8. 查看角色拥有的权限

```http
GET /api/admin/roles/2/permissions
Authorization: Bearer <admin_token>
```

返回该角色绑定的所有权限列表。

## 数据流

### 角色-用户-权限联动

```text
用户登录
  → 查询 user_roles（WHERE user_id = ?）
  → 获取 roleIDs
  → 查询 roles（WHERE id IN ? AND status = 1）
  → 提取 role codes → 写入 JWT.roles
  → 查询 role_permissions（WHERE role_id IN ?）
  → 获取 permissionIDs
  → 查询 permissions → 提取 codes → 写入 JWT.permissions
```

### 全量替换策略

```text
POST /api/admin/roles/:id/users
  → 校验角色存在
  → DELETE FROM user_roles WHERE role_id = ?
  → INSERT INTO user_roles (user_id, role_id) VALUES ...
```

## 安全约束

| 约束 | 说明 |
|---|---|
| admin 角色不可修改 | UpdateRole 检查 `role.Code == "admin"` |
| admin 角色不可删除 | DeleteRole 检查 `role.Code == "admin"` |
| 删除角色自动清理关联 | 删除 user_roles 中 role_id 匹配的记录 |
| 名称/编码唯一性 | 创建时检查 name OR code，修改时排除自己 |
| 分配用户/权限为全量替换 | 先 DELETE 再 INSERT，传空数组=清空 |

## 注意事项

- `code` 字段是角色的唯一标识，前端做权限判断时使用 code 而非 name
- 角色删除为硬删除，关联的 `user_roles` 记录同步清除
- 用户表中 `role` 字段是冗余字段（默认 "user"），实际 RBAC 以 `user_roles` 关联表为准
- 权限分配和菜单分配的逻辑与用户分配一致，均为全量替换
- 角色列表按 `sort asc, id asc` 排序，确保展示顺序可控

# 权限链路修改记录

## 文档目的

记录后台权限链路近期已经落地的关键修改，方便后续继续调整用户、角色、权限、菜单、API、组织数据范围时快速回看当前约定。

当前权限链路目标：

```text
API 管理
  -> API 绑定权限码
  -> 权限表沉淀权限码
  -> 角色绑定权限
  -> API 动态权限中间件拦截请求
  -> 菜单绑定权限码
  -> 菜单绑定 API
  -> 用户初始上下文返回可见菜单和权限码
```

新增接口：

```text
GET    /api/admin/apis
POST   /api/admin/apis
PUT    /api/admin/apis/:id
DELETE /api/admin/apis/:id
POST   /api/admin/apis/sync
POST   /api/admin/apis/sync-permissions
```

当前行为：

- `POST /api/admin/apis/sync` 从 Gin 已注册路由同步 API 记录。
- `POST /api/admin/apis/sync-permissions` 将 API 表中的权限码同步到 `permissions` 表。
- `method + path` 唯一。
- 公开接口不会同步成权限。

### 2. API 动态权限中间件

完成时间：2026-06-30

修改目标：

- 管理员接口从单纯 `HasRole("admin")` 改为基于 API 表的动态权限判断。
- 让 `apis.permission_code` 真正参与接口访问控制。

涉及文件：

```text
middleware/api_permission.go
router/router.go
docs/API管理.md
openspec/specs/api-management/spec.md
TODO.md
```

当前校验流程：

```text
请求进入 /api/admin
  -> JWTAuth 解析 token
  -> APIPermission 获取 method + c.FullPath()
  -> 查询 apis 表
  -> API 不存在则拒绝
  -> status != 1 则拒绝
  -> need_auth = 0 则放行
  -> admin 角色作为超级管理员兜底放行
  -> 普通管理员必须拥有 permission_code
```

首次初始化例外：

```text
POST /api/admin/apis/sync
POST /api/admin/apis/sync-permissions
```

这两个接口允许 `admin` 角色在 API 表未完整初始化时先执行，避免新库第一次启动后无法同步 API 元数据。

### 4. Token 实时失效

完成时间：2026-06-30

修改目标：

- 用户登录时将当前 `token_version` 写入 JWT。
- JWT 中间件每次请求校验 token 版本是否等于数据库当前版本。
- 权限、角色、菜单、组织成员、账号状态等影响登录态的变化发生后，立即提升相关用户 `token_version`。
- 旧 token 在下一次请求时返回 401，要求用户重新登录。

### 5. 用户强制下线

完成时间：2026-06-30

修改目标：

- 补齐管理员主动踢下线能力。
- 复用 `users.token_version` 机制，不新增会话表。
- 管理员执行强制下线后，目标用户旧 token 在下一次请求时立即失效。

新增接口：

```text
PUT /api/admin/users/:id/kick
```

当前行为：

- 管理员不能强制下线自己。
- 管理员不能强制下线其他管理员账号。
- 成功后只提升目标用户 `token_version`。
- 不修改用户资料，不禁用账号。

### 6. 配置迁移到 .env 与新 RBAC 超级管理员兜底

完成时间：2026-06-30

修改目标：

- 将数据库密码、Redis 密码、MinIO 密钥、JWT 密钥、超级管理员初始密码迁移到环境变量。
- 保留 `config.yaml` 的结构化配置能力，只通过 `*_env` 字段引用敏感环境变量。
- 启动时自动读取本地 `.env`，部署环境也可以直接注入同名环境变量。
- 数据库迁移后检查新 RBAC 体系中是否存在启用的超级管理员，没有则自动创建。
- 管理员身份判断彻底切到 `users -> user_roles -> roles.code = admin`。

涉及文件：

```text

docs/配置管理.md

```

敏感环境变量：

```text

REDIS_PASSWORD

MINIO_PASSWORD

```

超级管理员兜底规则：

- `roles.code = admin` 不存在时自动创建。
- 已经存在启用的 admin 角色用户时，不创建新账号，不重置密码。
- 不存在启用 admin 角色用户时，使用配置中的管理员信息和 `ADMIN_PASSWORD` 创建用户并绑定 admin 角色。
- 配置中的管理员用户名已存在但未绑定 admin 角色时，启动失败，避免误提权。
- 配置中的管理员用户已绑定 admin 角色但被禁用或软删除时，会恢复启用并提升 `token_version`。

### 8. 菜单与 API 联动

完成时间：2026-07-02

修改目标：

- 让菜单按钮和 API 使用同一套权限码。
- 新增 `menu_apis` 关联表，明确一个菜单按钮绑定了哪些 API。
- 支持菜单绑定 API。
- 支持查询菜单绑定 API。
- 支持从 API 生成按钮菜单。
- 修改已绑定 API 权限码时，同步更新关联菜单。
- 删除菜单或 API 时清理 `menu_apis`。

涉及文件：

```text

service/api.go

openspec/specs/menu-management/spec.md
openspec/specs/api-management/spec.md
TODO.md
```

新增表：

```text
menu_apis
```

新增接口：

```text
POST /api/admin/menus/:id/apis
GET  /api/admin/menus/:id/apis
POST /api/admin/apis/:id/menu-button
```

当前行为：

- 菜单绑定 API 时会校验菜单存在。
- 只允许绑定启用且需要认证的 API。
- 如果手动传 `permission_code`，菜单和绑定 API 都会统一使用该权限码。
- 如果不传 `permission_code`，系统会从 API 自动推导。
- 多个 API 权限码不一致时，需要手动传 `permission_code`。
- 从 API 生成按钮菜单时，会自动创建 `type=3` 菜单并写入 `menu_apis`。
- 系统会自动确保 `permissions` 表存在对应权限码。
- 联动修改会提升用户 `token_version`，旧 token 实时失效。

新增路由后需要执行：

```text
POST /api/admin/apis/sync
POST /api/admin/apis/sync-permissions
```

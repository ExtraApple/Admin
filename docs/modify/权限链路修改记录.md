# 权限链路修改记录

## 文档目的

记录近期围绕后台权限链路做过的修改，并列出下一步建议修改项。后续继续调整权限、菜单、API、角色相关逻辑时，可以在本文件追加记录。

当前权限链路目标：

```text
API 管理
  -> API 绑定权限码
  -> 角色绑定权限
  -> API 动态权限中间件拦截请求
  -> 菜单绑定权限码
  -> 用户初始上下文返回可见菜单和按钮
```

## 已完成修改

### 1. API 管理模块

完成时间：2026-06-30

修改目标：

- 维护系统中已注册接口的元数据。
- 支持从 Gin 路由自动同步 API。
- 支持 API 绑定权限码。
- 为后续动态权限中间件提供数据来源。

涉及文件：

```text
model/api.go
dto/api.go
service/api.go
service/api_helper.go
handler/api.go
router/router.go
docs/API管理.md
TODO.md
```

新增接口：

```text
GET    /api/admin/apis
POST   /api/admin/apis
PUT    /api/admin/apis/:id
DELETE /api/admin/apis/:id
POST   /api/admin/apis/sync
POST   /api/admin/apis/sync-permissions
```

核心字段：

```text
method
path
group
permission_code
status
need_auth
need_audit
```

当前行为：

- `POST /api/admin/apis/sync` 从 Gin 已注册路由同步 API 记录。
- `POST /api/admin/apis/sync-permissions` 将 API 表中的权限码同步到 `permissions` 表。
- `method + path` 唯一。
- 公开接口不会同步成权限。

注意事项：

- API 记录不是动态创建后端接口，真实接口仍然必须在 `router/router.go` 注册。
- `need_audit` 当前仍是元数据字段，尚未控制审计日志是否记录。

### 2. API 动态权限中间件

完成时间：2026-06-30

修改目标：

- 将管理员接口从单纯 `HasRole("admin")` 改成基于 API 表的动态权限判断。
- 让 `apis.permission_code` 真正参与接口访问控制。

涉及文件：

```text
middleware/api_permission.go
router/router.go
docs/API管理.md
TODO.md
```

当前校验流程：

```text
请求进入 /api/admin
  -> JWTAuth 解析 token
  -> APIPermission 获取 method + c.FullPath()
  -> 查询 apis 表
  -> API 不存在则拒绝
  -> status != 1 则拒绝
  -> need_auth = 0 则放行
  -> admin 角色作为超级管理员兜底放行
  -> 普通管理角色必须拥有 permission_code
```

首次初始化例外：

```text
POST /api/admin/apis/sync
POST /api/admin/apis/sync-permissions
```

这两个接口允许 `admin` 角色在 API 表未完整初始化时先执行，避免新库第一次启动后无法同步 API 元数据。

注意事项：

- 普通管理角色访问接口时，必须拥有 API 绑定的权限码。
- 如果 API 未绑定权限码，普通管理角色会被拒绝。
- `admin` 角色目前是超级管理员兜底。

### 3. 菜单绑定权限码和按钮权限

完成时间：2026-06-30

修改目标：

- 菜单记录支持绑定 `permission_code`。
- 页面菜单和按钮菜单可以根据权限码控制可见性。
- `/api/user/context` 返回的菜单树只包含当前用户可见菜单。

涉及文件：

```text
model/menu.go
dto/menu.go
service/menu.go
service/menu_helper.go
docs/菜单管理.md
TODO.md
```

新增字段：

```text
menus.permission_code
```

当前过滤规则：

```text
admin 角色：返回全部启用菜单
普通用户：只返回角色绑定菜单
permission_code 为空：角色绑定后可见
permission_code 不为空：用户必须拥有对应权限码
父级目录：只要下面存在可见子菜单或按钮，就保留
```

按钮菜单约定：

```text
type = 1 目录
type = 2 页面菜单
type = 3 按钮权限
```

按钮菜单通常没有前端路由，所以 `menus.path` 已调整为可空，避免多个按钮菜单因空字符串触发唯一索引冲突。

示例按钮：

```json
{
  "parent_id": 2,
  "name": "删除用户",
  "path": "",
  "component": "",
  "icon": "",
  "permission_code": "admin.users.id.delete",
  "sort": 1,
  "type": 3,
  "status": 1
}
```

### 5. 权限变更后的 Token 实时失效

完成时间：2026-06-30

修改目标：

- 用户登录时把当前 `token_version` 写入 JWT。
- JWT 中间件每次请求都校验 token 中版本是否等于数据库当前版本。
- 权限、角色、菜单、组织成员、账号状态等影响登录态的变化发生后，立即提升相关用户 `token_version`。
- 旧 token 在下一次请求时立即返回 401，要求用户重新登录。

涉及文件：

```text
model/user.go
utils/jwt.go
middleware/jwt.go
service/token_version.go
service/user.go
service/admin_user.go
service/role.go
service/permission.go
service/menu.go
service/organization.go
```

新增字段：

```text
users.token_version
```

JWT 新增字段：

```text
token_version
```

当前失效触发点：

```text
用户修改密码
管理员修改用户信息
管理员禁用/解禁用户
管理员强制用户下线
角色修改
角色删除
角色分配用户
角色绑定权限
权限删除
菜单创建/修改/删除
角色绑定菜单
前端菜单同步
组织成员绑定
```

当前校验流程：

```text
JWTAuth 解析 token
  -> 校验 Redis 黑名单
  -> 读取 claims.token_version
  -> 查询 users.token_version
  -> 版本一致则放行
  -> 版本不一致则返回 401 "Token已失效，请重新登录"
```

注意事项：

- 历史用户如果 `token_version <= 0`，登录时会自动初始化为 `1`。
- 用户被禁用后，即使 token 版本未变化，也会在 JWT 中间件中被拒绝。
- 删除用户后，中间件查询不到用户，也会拒绝旧 token。

### 6. 用户强制下线

完成时间：2026-06-30

修改目标：

- 补齐管理员主动踢下线能力。
- 复用现有 `users.token_version` 机制，不新增会话表。
- 管理员执行强制下线后，目标用户旧 token 在下一次请求时立即失效。

涉及文件：

```text
service/admin_user.go
handler/admin_user.go
router/router.go
docs/用户管理.md
TODO.md
```

新增接口：

```text
PUT /api/admin/users/:id/kick
```

当前行为：

- 管理员不能强制下线自己。
- 管理员不能强制下线其他管理员账号。
- 目标用户不存在时返回错误。
- 成功后只提升目标用户 `token_version`，不修改用户资料、不禁用账号。
- 目标用户旧 access token 和 refresh token 都会因为 `token_version` 不匹配而失效。

权限同步提醒：

```text
POST /api/admin/apis/sync
POST /api/admin/apis/sync-permissions
```

同步后，普通管理角色需要拥有该接口生成的权限码才能调用；`admin` 角色仍可作为超级管理员兜底访问。

### 7. 配置迁移到 .env 与新 RBAC 超级管理员兜底

完成时间：2026-06-30

修改目标：

- 将数据库密码、Redis 密码、MinIO 密钥、JWT 密钥、超级管理员初始密码从 `config.yaml` 迁移到环境变量。
- 保留 `config.yaml` 的结构化配置能力，只通过 `*_env` 字段引用敏感环境变量。
- 启动时自动读取本地 `.env`；部署环境也可以直接注入同名环境变量，不强依赖 `.env` 文件。
- 在数据库迁移后检查新 RBAC 体系中是否存在启用的超级管理员，没有则自动创建。
- 管理员身份判断彻底切到 `users -> user_roles -> roles.code = admin`，不再依赖 `users.role = admin`。

涉及文件：

```text
config.yaml
.env.example
.gitignore
initialize/server.go
initialize/admin.go
main.go
service/admin_user_helper.go
docs/配置管理.md
docs/README.md
README.md
openspec/specs/rbac/spec.md
```

当前敏感环境变量：

```text
MYSQL_PASSWORD
REDIS_PASSWORD
MINIO_USERNAME
MINIO_PASSWORD
JWT_SECRET
ADMIN_PASSWORD
```

超级管理员兜底规则：

- `roles.code = admin` 不存在时自动创建。
- 已经存在启用的 admin 角色用户时，不创建新账号、不重置密码。
- 不存在启用 admin 角色用户时，用配置中的 `admin.username/email/nickname` 和 `ADMIN_PASSWORD` 创建用户并绑定 admin 角色。
- 配置中的管理员用户名已存在但未绑定 admin 角色时，启动失败，避免误提权。
- 配置中的管理员用户已绑定 admin 角色但被禁用或软删除时，会恢复启用并提升 `token_version`。

## 下一步建议修改

### 1. 数据权限 / 组织数据范围

当前问题：

- 组织管理已经有组织树和成员绑定。
- 但还没有控制用户能看哪些组织或数据。

建议增加角色数据范围：

```text
全部数据
本组织数据
本组织及下级组织数据
仅本人数据
自定义组织数据
```

可能新增字段：

```text
roles.data_scope
role_data_scopes
```

优先级：高

原因：

- 后台管理系统常见需求是不同管理员只能管理不同组织范围内的数据。

### 2. 菜单与 API 权限联动优化

当前问题：

- 菜单可以绑定权限码。
- API 也可以绑定权限码。
- 但菜单和 API 之间没有直接联动关系。

建议：

```text
菜单按钮 permission_code 与 API permission_code 使用同一套权限码
在 API 管理页面选择权限码
在菜单按钮页面选择同一个权限码
前端按钮可见性和后端接口权限保持一致
```

可选增强：

```text
API 管理页面增加“生成按钮菜单”操作
菜单管理页面选择 API 后自动带出 permission_code
```

优先级：中高

### 3. 文件上传安全策略

当前问题：

- 文件管理已经支持上传、下载、浏览、轮转。
- 但作为后台系统，还需要更明确的安全限制。

建议补充：

```text
文件大小限制
MIME 类型白名单
图片格式校验
用户或组织空间配额
危险扩展名拒绝
文件访问权限控制
```

优先级：中

### 4. 后台首页仪表盘

当前问题：

- 后台管理功能很多，但缺少进入后台后的总览页。

建议指标：

```text
用户总数
今日登录数
今日操作数
异常登录次数
文件上传量
最近操作日志
待处理消息
系统运行状态
```

优先级：中

### 5. 内部消息模块

```text
1. 组织数据范围 / 数据权限
2. 菜单与 API 权限联动优化
3. 文件上传安全策略
4. 后台首页仪表盘
```

如果只选一个最优先项，建议先做：

```text

```

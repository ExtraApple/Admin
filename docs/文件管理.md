# 文件管理实施文档

> **文档定位**: 已实现模块的实现说明。当前文件上传、列表、详情、删除、浏览和轮转行为以 `openspec/specs/file-management/spec.md` 为准；本文保留 MinIO、数据模型和测试步骤说明。

## 当前状态

文件管理已完成完整的文件上传、查看、修改、删除和浏览功能，基于 MinIO 对象存储，所有接口为管理员接口。

## 模块目标

文件管理负责整个后台系统的文件上传、存储、检索和生命周期管理，包括文件元数据持久化和文件轮转归档。

最终效果：

```text
管理员上传文件
  → 校验大小（max 50MB）
  → 生成 UUID 对象名
  → 上传到 MinIO（bucket: files）
  → 写入 files 表记录元数据
  → 返回文件信息 + 下载链接
```

## 功能清单

| 功能 | 状态 | 说明 |
|---|---|---|
| 上传文件 | ✅ | MinIO 存储，UUID 命名，大小限制 50MB |
| 文件列表 | ✅ | 分页查询，支持按对象名前缀筛选 |
| 文件详情 | ✅ | 元数据 + 预签名下载链接（300s 有效） |
| 修改文件名 | ✅ | 只修改数据库 name 字段，MinIO 对象名不变 |
| 删除文件 | ✅ | MinIO 对象 + DB 记录双删，不可恢复 |
| 按目录浏览 | ✅ | 直接浏览 MinIO 对象列表 |
| 文件轮转 | ✅ | 热存储 → 冷存储，定时任务驱动，可配置开关 |

## 数据模型设计

### model/file.go

```go
// File 文件管理表 — 存储文件元数据，实际文件在 MinIO
type File struct {
    gorm.Model
    Name        string `gorm:"type:varchar(255);comment:原始文件名"`
    Bucket      string `gorm:"type:varchar(100);not null;comment:MinIO bucket"`
    ObjectName  string `gorm:"type:varchar(500);not null;comment:MinIO 对象名"`
    ContentType string `gorm:"type:varchar(100);comment:MIME 类型"`
    Size        int64  `gorm:"comment:文件大小(字节)"`
    UploaderID  uint   `gorm:"index;comment:上传者ID"`
}
```

## 接口设计

所有接口在管理员路由组下：

```text
/api/admin
```

需要：

```text
Authorization: Bearer <admin_token>
```

### 1. 上传文件

```http
POST /api/admin/files
Content-Type: multipart/form-data
Authorization: Bearer <admin_token>
```

Body：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| file | file | 是 | 选择本地文件，最大 50MB |

成功返回：

```json
{
  "code": 200,
  "msg": "上传成功",
  "data": {
    "id": 1,
    "name": "report.pdf",
    "bucket": "files",
    "object_name": "a1b2c3d4-e5f6-7890-abcd-ef1234567890.pdf",
    "content_type": "application/pdf",
    "size": 1048576,
    "uploader_id": 3,
    "created_at": "2026-06-26 10:00:00"
  }
}
```

说明：

- 原始文件名保存在 `name`
- MinIO 中实际保存的是 UUID 命名的对象名（带原始扩展名）
- 文件元数据同步写入数据库 `files` 表
- 上传失败时会清理已上传的 MinIO 对象（回滚）

### 2. 文件列表

```http
GET /api/admin/files?page=1&size=10&prefix=
Authorization: Bearer <admin_token>
```

Query 参数：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| page | int | 1 | 页码 |
| size | int | 10 | 每页条数 |
| prefix | string | 空 | 按对象名前缀筛选 |

成功返回：

```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": 1,
        "name": "report.pdf",
        "bucket": "files",
        "object_name": "a1b2c3d4.pdf",
        "content_type": "application/pdf",
        "size": 1048576,
        "uploader_id": 3,
        "created_at": "2026-06-26 10:00:00"
      }
    ],
    "total": 50,
    "page": 1,
    "size": 10
  }
}
```

说明：

- 基于数据库的 `files` 表分页查询
- `prefix` 筛选基于 `object_name LIKE 'prefix%'`
- 按 `created_at desc` 降序排列（最新在前）

## 文件轮转设计

### service/file_rotation.go

```go
func RotateFiles(conf *initialize.Config) {
    if !conf.FileRotation.Enabled {
        return // 未启用
    }

    hot := conf.FileRotation.HotBucket   // "files"
    cold := conf.FileRotation.ColdBucket  // "files-archive"
    days := conf.FileRotation.Days        // 30
    batch := conf.FileRotation.BatchSize  // 100

    // 查询超过天数的文件
    cutoff := time.Now().AddDate(0, 0, -days)
    var files []model.File
    global.DB.Where("created_at < ? AND bucket = ?", cutoff, hot).
        Limit(batch).Find(&files)

    // 逐文件移动
    for _, f := range files {
        // MinIO: hot bucket → cold bucket
        utils.MoveFile(hot, cold, f.ObjectName)

        // 更新数据库: bucket = cold
        global.DB.Model(&f).Update("bucket", cold)
    }
}
```

### 配置 (config.yaml)

```yaml
file_rotation:
  enabled: false          # 是否启用文件轮转
  days: 30                # 超过多少天触发轮转
  hot_bucket: "files"      # 热存储 bucket
  cold_bucket: "files-archive"  # 冷存储 bucket
  batch_size: 100          # 每轮处理数量上限
```

### 启动方式 (main.go)

```go
if conf.FileRotation.Enabled {
    go func() {
        ticker := time.NewTicker(24 * time.Hour) // 每天执行一次
        defer ticker.Stop()
        for range ticker.C {
            service.RotateFiles(conf)
        }
    }()
}
```

## 数据流

### 上传流程

```text
POST /api/admin/files (multipart/form-data)
  → 校验文件大小 ≤ 50MB
  → 提取原始文件扩展名
  → utils.UploadStream(bucket="files", prefix="", ext, contentType, reader, size)
      → 生成 UUID 对象名: uuid.ext
      → MinIO PutObject
  → DB.Create(&File{name, bucket, objectName, contentType, size, uploaderID})
  → 返回 FileInfo
```

### 删除流程

```text
DELETE /api/admin/files/:id
  → 查询 files 表
  → MinIO RemoveObject(bucket, objectName)
  → DB.Unscoped().Delete(&file)  // 硬删除
```

### 文件轮转流程

```text
每天定时触发
  → 查询 files WHERE created_at < cutoff AND bucket = hotBucket (LIMIT batch)
  → 逐文件：
      → MinIO: CopyObject(hot → cold) + RemoveObject(hot)
      → DB: UPDATE bucket = coldBucket
      → 失败回滚：移回热存储
```

## 安全约束

| 约束 | 说明 |
|---|---|
| 仅管理员操作 | 所有文件接口在 admin 路由组下 |
| 文件大小限制 | 上传最大 50MB |
| Bucket 固定 | 统一使用 `files` bucket，不可用户指定 |
| 删除不可恢复 | MinIO + DB 双删，操作需谨慎 |
| 下载链接有效期 | 300 秒（5 分钟），防止链接泄露长期有效 |

## 注意事项

- bucket 名固定为 `files`，不可通过接口参数指定（安全考虑）
- 上传文件使用 UUID 命名避免冲突和文件遍历
- 原始文件名保留在 `name` 字段，UUID 对象名保留在 `object_name`
- 文件列表建议优先用数据库分页接口（`/api/admin/files`），按目录浏览（`/api/admin/files-browse`）更适合探索 MinIO 的实际存储结构
- 上传失败会自动清理 MinIO 中的残留对象（事务回滚）
- 文件轮转默认关闭，需要通过配置文件 `file_rotation.enabled: true` 开启
- 轮转是单文件逐个处理，失败不影响其他文件（带回滚机制）
- 头像上传使用独立的 `image` bucket 和 `CleanOldFiles` 机制（保留最近 2 个），与文件管理的 `files` bucket 隔离

# 操作日志实施文档

> **文档定位**: 已实现说明。正式行为在完成 OpenSpec 归档后以 `openspec/specs/logging/spec.md` 为准。

## 当前状态

> **状态**: 已实现基础版 — 已支持自动记录 API 审计日志和管理员查询。

操作日志是后台审计基础设施，通过 Gin 中间件拦截 `/api/*` 请求，自动记录操作轨迹。

## 模块目标

记录每一次 API 调用的核心信息：

```text
谁、什么时候、调了哪个接口、什么参数、返回状态、耗时多久、来源 IP
```

## 当前已实现功能

- [x] API 调用日志记录
- [x] 登录日志查询
- [x] 操作日志查询
- [x] 权限变更日志查询
- [x] 数据访问日志查询
- [x] JSON 请求体敏感字段脱敏
- [x] multipart 上传请求跳过文件内容记录
- [x] 日志冷热归档策略

## 数据模型

表名：`audit_logs`

```go
type AuditLog struct {
    ID        uint
    UserID    uint
    Username  string
    Method    string
    Path      string
    Query     string
    Body      string
    Status    int
    Duration  int64
    ClientIP  string
    UserAgent string
    Category  string
    CreatedAt time.Time
}
```

冷表：`audit_log_archives`

冷表字段和热表基本一致，额外增加：

```go
ArchivedAt time.Time
```

## 日志分类

| category | 说明 |
|---|---|
| `api` | 默认 API 日志 |
| `login` | 登录接口日志 |
| `operation` | POST / PUT / DELETE 增删改日志 |
| `permission` | 权限、菜单、角色绑定关系相关写操作 |
| `data_access` | GET 查询类日志 |

分类优先级：

```text
POST /api/login
  → login

GET 任意 /api/*
  → data_access

POST/PUT/DELETE /api/admin/permissions*
POST/PUT/DELETE /api/admin/permission-groups*
POST/PUT/DELETE /api/admin/menus*
POST/PUT/DELETE /api/admin/roles/:id/permissions
POST/PUT/DELETE /api/admin/roles/:id/menus
POST/PUT/DELETE /api/admin/roles/:id/users
  → permission

其他 POST/PUT/DELETE/PATCH
  → operation

其他请求
  → api
```

注意：

- `GET /api/admin/roles` 是数据访问日志，不是权限变更日志。
- `POST /api/admin/roles` 是普通操作日志，不是权限变更日志。
- `POST /api/admin/roles/:id/permissions` 才是权限变更日志。

## 敏感字段脱敏

JSON body 中以下字段会被记录为 `***`：

- `password`
- `old_password`
- `new_password`
- `confirm_password`
- `captcha_code`
- `access_token`
- `refresh_token`
- `token`

上传文件接口不会记录文件内容，只记录：

```text
[multipart omitted]
```

## 已实现接口

所有接口都需要管理员 token。

### 1. 查询全部 API 审计日志

```http
GET /api/admin/audit-logs?page=1&size=10
Authorization: Bearer <admin_token>
```

可选查询参数：

| 参数 | 示例 | 说明 |
|---|---|---|
| `page` | `1` | 页码 |
| `size` | `10` | 每页数量 |
| `user_id` | `3` | 操作用户 ID |
| `method` | `POST` | 请求方法 |
| `path` | `/api/admin/users` | 路径模糊查询 |
| `status` | `200` | HTTP 状态码 |
| `category` | `operation` | 日志分类 |
| `start_time` | `2026-06-01 00:00:00` | 开始时间 |
| `end_time` | `2026-06-30 23:59:59` | 结束时间 |

### 2. 查询登录日志

```http
GET /api/admin/login-logs?page=1&size=10
Authorization: Bearer <admin_token>
```

### 3. 查询操作日志

```http
GET /api/admin/operation-logs?page=1&size=10
Authorization: Bearer <admin_token>
```

返回 `operation` 和 `permission` 两类日志。

### 4. 查询权限变更日志

```http
GET /api/admin/permission-logs?page=1&size=10
Authorization: Bearer <admin_token>
```

### 5. 查询数据访问日志

```http
GET /api/admin/data-access-logs?page=1&size=10
Authorization: Bearer <admin_token>
```

## Apifox 测试顺序

1. 启动服务，让自动迁移创建 `audit_logs` 表。
2. 调用 `GET /api/captcha`，确认产生一条 API 日志。
3. 调用 `POST /api/login`，确认产生一条登录日志。
4. 调用任意管理员增删改接口，确认操作日志可查询。
5. 调用角色绑定权限、角色绑定菜单、角色绑定用户接口，确认权限变更日志可查询。
6. 调用任意 GET 查询接口，确认数据访问日志可查询。
7. 测试登录 body，确认密码和验证码已脱敏。
8. 测试文件上传，确认 body 为 `[multipart omitted]`。

## 冷热归档配置

```yaml
audit_log_archive:
  enabled: false
  retention_days: 90
  batch_size: 1000
```

说明：

| 配置 | 作用 |
|---|---|
| `enabled` | 是否启动归档任务 |
| `retention_days` | 热表保留最近多少天 |
| `batch_size` | 每次最多归档多少条 |

归档流程：

```text
每天执行一次
  → 查询 audit_logs 中 created_at < now - retention_days 的记录
  → 批量写入 audit_log_archives
  → 写入成功后删除 audit_logs 中对应记录
```

当前归档任务默认关闭。需要启用时，把 `enabled` 改为 `true`。

## 注意事项

- 公开接口没有 JWT 上下文时，`user_id` 会记录为 `0`。
- 当前 `username` 暂未从 JWT 中注入，默认可能为空。
- 审计日志查询接口本身也会被记录。
- 异步写入失败不会影响业务响应。
- 当前采用冷热表归档方案，暂不做按月分表。

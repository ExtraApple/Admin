# 权限管理实施文档

## 当前状态

权限管理已完成全部 CRUD、权限分组管理、自动同步路由权限、以及基于 JWT 的中间件拦截功能。

## 模块目标

权限管理维护系统中的权限码定义，配合 `HasPermission` 中间件实现接口级别的权限控制，配合前端实现按钮级别的权限控制。

最终效果：

```text
用户请求接口
  → JWT 中间件：解析 Token，注入 permissions 到 Context
  → HasPermission("user:delete") 中间件：检查 Context 中的 permissions 是否包含所需权限码
  → 通过 → 执行业务逻辑
  → 拒绝 → 403 Forbidden
```

## 数据模型设计

### model/permission.go

```go
// Permission 权限表 — 存储权限码（如 "user:delete"）
type Permission struct {
    gorm.Model
    Name  string `gorm:"type:varchar(100);not null;comment:权限名称"`
    Code  string `gorm:"type:varchar(100);not null;uniqueIndex;comment:权限码"`
    Group string `gorm:"type:varchar(50);comment:权限分组"`
    Sort  int    `gorm:"type:int;default:0;comment:排序"`
}

// RolePermission 角色-权限关联表
type RolePermission struct {
    RoleID       uint `gorm:"primaryKey;comment:角色ID"`
    PermissionID uint `gorm:"primaryKey;comment:权限ID"`
}

// PermissionGroup 权限分组表
type PermissionGroup struct {
    gorm.Model
    Name string `gorm:"type:varchar(50);not null;uniqueIndex;comment:分组名称"`
    Sort int    `gorm:"type:int;default:0;comment:排序"`
}
```

## HasPermission 中间件

### middleware/admin.go

```go
// HasPermission 校验当前用户是否拥有指定权限码之一（OR 逻辑）
func HasPermission(allowed ...string) gin.HandlerFunc {
    return func(c *gin.Context) {
        perms, _ := c.Get("permissions")
        permList, ok := perms.([]string)
        if !ok {
            c.AbortWithStatusJSON(403, gin.H{"code": 403, "msg": "无操作权限"})
            return
        }
        for _, p := range permList {
            for _, a := range allowed {
                if p == a {
                    c.Next()
                    return
                }
            }
        }
        c.AbortWithStatusJSON(403, gin.H{"code": 403, "msg": "无操作权限"})
    }
}
```

### 使用方式

```go
// 单权限：必须拥有 user:list
admin.GET("/users").Use(HasPermission("user:list"))

// 多权限：满足其一即可（OR 逻辑）
articles.DELETE("/:id").Use(HasPermission("article:delete", "article:manage"))
```

## JWT 中的权限

登录时自动查询 `user → user_roles → role_permissions → permissions`，将所有权限码写入 JWT：

```go
// service/permission.go → GetUserPermissions
func GetUserPermissions(userID uint) []string {
    // JOIN user_roles + role_permissions 查询
    // 返回去重后的权限码数组
}
```

JWT Payload：

```json
{
  "user_id": 3,
  "roles": ["admin"],
  "permissions": ["user:list", "user:delete", "role:create", "role:update"]
}
```

## 权限码命名规范

```text
{模块}:{操作}

示例：
  user:list       用户列表
  user:create     创建用户
  user:update     修改用户
  user:delete     删除用户
  role:create     创建角色
  role:update     修改角色
  article:create  创建文章
  article:delete  删除文章
```

## 参与角色管理的权限码

权限不直接绑定用户，而是通过角色中转：

```text
用户 → user_roles → 角色 → role_permissions → 权限码 → JWT → HasPermission 中间件
```

## 权限 vs 菜单

权限和菜单是两套独立系统，应配合使用：

```text
菜单控制"看不看得到"
权限控制"能不能执行"
```

示例：

```text
editor 角色：
  菜单：文章管理              → 侧边栏显示
  权限：article:list          → 可以进入列表页
        article:create        → 显示"新建"按钮
        article:update        → 显示"编辑"按钮
        （无 article:delete） → 不显示"删除"按钮
```

## 同步权限规则

- 跳过 `/ping`、`/api/captcha`、`/api/register`、`/api/login`
- 只同步 `/api/` 开头的路由
- 自动生成 code：路径中的 `/api/` 替换掉，`/` 替换为 `.`，`:id` 替换为 `id`，追加 `.method`
- 已存在的 code 不会重复创建
- 分组自动取 code 的第一段

## 注意事项

- 权限码 `code` 创建后不可修改（已在 UpdatePermissionReq 中排除）
- 删除权限时自动清理 `role_permissions` 关联记录
- `GetUserPermissions` 使用 JOIN 查询提高性能
- `SyncPermissions` 需要传入 `*gin.Engine`，通过 `r.Routes()` 获取所有已注册路由
- `HasPermission` 中间件使用 OR 逻辑（传入多个权限码时满足其一即可）
- 权限缓存于 JWT 中，权限变更后需重新登录才能生效
- `permission-codes` 接口返回扁平数组，按 group + sort 排序，方便前端做权限判断
